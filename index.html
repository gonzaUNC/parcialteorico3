Frameworks

Core Concept

Inversión de Control (IoC)

"No nos llames, nosotros te llamamos". Flask decide cuándo invocar tu código en respuesta a una request HTTP entrante. Vos no hacés el bucle infinito escuchando el puerto, el framework lo hace.

Estructura

Hot Spots

Los puntos donde el desarrollador introduce la lógica específica de la aplicación. Ej: La función crear_usuario() decorada con @app.route. Es tu código personalizado.

Estructura

Frozen Spots

El código interno del framework que NO modificás ni reimplementás. Ej: El mecanismo de ruteo de Flask (recibir HTTP, parsear URL y despachar).

from flask import Flask, request, jsonify
app = Flask(__name__)

# ESTO ES LÓGICA DE NEGOCIO (No es ni frozen ni hot spot per se, es tu librería)
def validar_usuario(datos): ...

# ESTO ES UN HOT SPOT (El framework va a llamar a esta función)
@app.route("/usuarios", methods=["POST"])
def crear_usuario():
    datos = request.get_json()
    if not validar_usuario(datos):
        return jsonify({"error": "datos incompletos"}), 400
    return jsonify({"id": 1}), 201

Programación Defensiva / Seguridad

Concepto Parcial

Programación Ofensiva

Fallar rápido y ruidosamente. Un assert aborta el programa inmediatamente si las condiciones no se dan, en lugar de intentar arreglarlo y arrastrar un error silencioso. El código del parcial ES programación ofensiva.

Concepto Parcial

Defensa por Contrato

El uso de asserts al inicio de una función para validar precondiciones (ej: la lista no está vacía) antes de modificar cualquier estado.

Diferencia Clave

Guardas vs Excepciones

Los asserts actúan como guardas (previenen la ejecución si no se cumple la condición). NO son equivalentes a mecanismos de manejo de excepciones (try/catch), que buscan recuperarse del error.

def calcular_promedio(valores):
    # Estos asserts son GUARDAS y DEFENSA POR CONTRATO (Precondiciones). 
    # Si fallan crashean el programa: PROGRAMACIÓN OFENSIVA.
    assert isinstance(valores, list), "valores debe ser una lista"
    assert len(valores) > 0, "la lista no puede estar vacía"
    return sum(valores) / len(valores)

Scripting

Propiedad del lenguaje de Scripting ¿Aparece en el código del parcial? (Python)
Las variables no requieren asignación explícita de tipo. SÍ. En la línea resultado = {} o datos = procesar_archivo(...) no se declara Dict datos = ....
Seguridad de tipos (evita comportamientos inesperados) NO en este fragmento. Python permite meter strings y números en el mismo diccionario dinámicamente sin quejarse hasta que explota en runtime.
Buena comunicación con el S.O. NO en este fragmento. Abrir un archivo (`open()`) es estándar en cualquier lenguaje, no invoca comandos de consola (como haría `os.system()` o bash).
Reglas de alcance explícitas NO. En Python el scope es implícito (local a la función, no requiere usar palabras clave como `let` o `local` para declarar el ámbito).

Prolog: Árbol de Backtracking

Consulta: ?- equipo(elena, ana).

1. Buscar regla: equipo(elena, ana) :- departamento(elena, ana), not(jefe(elena, ana)).
2. Subobjetivo 1: Resolver departamento(elena, ana) (TRUE)
Regla: departamento(elena, ana) :- especialista(elena, A), especialista(ana, A).
- Busca especialista(elena, A) -> Encuentra hecho especialista(elena, traumatologia). Unifica A = traumatologia.
- Busca especialista(ana, traumatologia) -> Encuentra hecho. Verdadero.
3. Subobjetivo 2: Resolver not(jefe(elena, ana)) (TRUE)
Para resolver un NOT, Prolog intenta probar lo de adentro. Si falla, el NOT es verdadero (Negation as Failure).
- Intenta probar jefe(elena, ana)
Regla: jefe(elena, ana) :- deriva(elena, ana), departamento(elena, ana).
- Busca deriva(elena, ana) -> Revisa los hechos. Solo existen derivaciones desde 'clara' y 'esteban'. NO hay desde 'elena'. FALLA.
Como deriva falló, jefe falla.
Como jefe(elena, ana) falló, not(jefe(elena, ana)) es VERDADERO.
Conclusión: Ambos subobjetivos de la regla `equipo` son verdaderos.
Resultado final: TRUE

Concurrencia

Línea de Scala Concepto Aplicado Explicación Bosteril
val t = new Thread(() => ...)
t.start()		 Manejo explícito de procesos (hilos) Estás instanciando la clase Thread y dándole start() a mano. Vos controlás el ciclo de vida del hilo directamente a lo Picapedrero.
val f1 = Future { ... } Ejecución concurrente El bloque de código adentro del Future se manda a ejecutar de forma asíncrona en un pool de hilos (ExecutionContext) mientras el programa principal sigue de largo.
Future.sequence(List(f1, f2)) Sincronización y Ej. Concurrente Combina múltiples futuros. Actúa como una barrera de Sincronización lógica (el Future resultante se completa cuando *ambos* f1 y f2 terminan).
t.join() Sincronización El hilo principal frena y espera a que el hilo t termine. Sincroniza la ejecución de dos ramas de control.
Await.result(...) Sincronización Bloquea el hilo actual hasta que el Future devuelva el valor (o pase el timeout de 5s).

Múltiple Choice Bosteril

1. En Flask, `crear_usuario` es un hotspot: es el punto donde el desarrollador introduce la lógica específica de la app.
2. En Flask, el desarrollador decide cuándo se llama a `crear_usuario` invocándolo explícitamente cuando llega la request.
3. El uso de múltiples `asserts` para validar tipos antes de ejecutar la lógica es un ejemplo de programación ofensiva.
4. En Prolog, para demostrar `equipo(elena, ana)`, el motor NO comprueba `departamento(ana, elena)`.
5. Usar `t.join()` en programación concurrente es un mecanismo de Exclusión Mutua.
