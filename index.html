Tema 1

Programación Lógica (Prolog)

A diferencia de la programación imperativa, donde la primitiva básica es la función F(x)=y, en programación lógica la primitiva básica es la relación R(x,y): simplemente declaramos que la relación R se da entre x e y. El motor resuelve cómo demostrarlo.

Motor

Inferencia y Backtracking

Prolog no ejecuta línea por línea. Resuelve objetivos buscando una prueba lógica. Si una rama falla, retrocede (backtracking) automáticamente e intenta la siguiente cláusula disponible. Es una búsqueda en profundidad dirigida por el objetivo.

Estructura

Hechos y Reglas

Los hechos son verdades axiomáticas (mago(harry).). Las reglas son implicaciones lógicas (h :- p1, p2. = "h es cierto si p1 y p2 son ciertos"). Un programa Prolog es un conjunto de hechos y reglas.

Términos

Variables, Constantes y Estructuras

Las variables empiezan en mayúscula (X, Harry). Las constantes son enteros o átomos (24, zebra, 'Bob'). Las estructuras son predicados con argumentos (habla(Y, castellano)). El comodín _ marca lo irrelevante.

Lógica

Cláusulas de Horn

Una cláusula de Horn tiene una cabeza h y un cuerpo de predicados: h ← p1, p2, …, pn. Cualquier cláusula se puede expresar como ¬p1 ∨ ¬p2 ∨ … ∨ h. No todo predicado lógico es expresable como cláusula de Horn (ej: disyunciones en la conclusión).

Unificación

La unificación es el proceso de igualar dos términos encontrando una sustitución de variables que los haga idénticos. Es el corazón del pattern-matching de Prolog.

% Ejemplos de Unificación
f(X)   unifica con f(3)    → [X=3]
f(f(Y)) unifica con f(X)   → [X=f(Y)]
g(X,Y) NO unifica con f(3) → falla

% Variables lógicas
?- nonstop(okc, X).
X = phx ;
No

?- nonstop(Y, dal).
Y = aus ;
No
💡

Instanciación vs. Asignación

En Prolog, cuando una variable se unifica con un valor durante la resolución se llama instanciación. No es una asignación imperativa — si la rama falla, la instanciación se deshace automáticamente (backtracking).

No-Determinismo y Consultas Múltiples

Los predicados pueden devolver múltiples respuestas (con ; para pedir la siguiente) o ninguna. Esto es el no-determinismo de Prolog.

% No-determinismo: múltiples resultados
?- nonstop(dal, X).
X = hou ;
X = okc ;
No

% Conjunción lógica (coma = AND)
?- nonstop(aus, X), nonstop(X, okc).
X = dal ;
X = hou ;
No

% Recursión: llegar desde un nodo a otro
llegar(X, X).
llegar(X, Z) :- nonstop(X, Y), llegar(Y, Z).

?- llegar(aus, phx).
Yes

El Operador Cut (!)

El cut (!) fuerza a Prolog a no revisar alternativas una vez que ya encontró un resultado. Hay dos tipos:

Green Cut

Cut de optimización

No cambia el significado lógico del programa, solo evita trabajo innecesario. Ej: max(X,Y,X) :- X > Y, !. — si X > Y ya sabemos que X es el máximo, no hay que seguir buscando.

Red Cut

Cut semántico

Cambia el comportamiento lógico del programa. Si se elimina, el programa da resultados diferentes. Se usa con más cuidado porque puede hacer el código difícil de razonar.

% Negación implementada con cut y fail
not(G) :- G, !, fail.
not(_).
% "fail" es un objetivo especial que siempre falla
% Si G tiene éxito, el cut evita que not tenga éxito, y fail lo confirma
% Si G falla, la segunda cláusula not(_) tiene éxito

% Ejemplo de uso de not en el parcial:
equipo(X, Y) :- departamento(X,Y), not(jefe(X, Y)).

Listas en Prolog

% Estructura de listas
[]            % lista vacía
[1, 2, 3]     % lista con 3 elementos
[H|T]         % H = cabeza, T = cola (sublista)
[_, X, Y]     % _ = elemento cualquiera (comodín)

% append: concatenar dos listas
append([], X, X).
append([Head|Tail], Y, [Head|Z]) :- append(Tail, Y, Z).

% member: pertenencia
member(X, [X|_]).
member(X, [_|Y]) :- member(X, Y).

% Prefijos y sufijos
prefix(X, Z) :- append(X, _, Z).
suffix(Y, Z) :- append(_, Y, Z).
⚠️

Completitud: Prolog es incompleto

El procedimiento de búsqueda de Prolog devuelve solo cosas verdaderas, pero no puede probar todo lo que es verdadero. Además, un cambio en el orden de las cláusulas puede causar un bucle infinito (ej: usar recursión a izquierda en lugar de a derecha).

Tema 2

Frameworks e Inversión de Control

Un framework es una abstracción de software reusable que provee funcionalidades genéricas instanciables en software específico. Es la evolución del boilerplate code. Se diferencia de una librería por el principio de Inversión de Control (IoC).

IoC

Inversión de Control

El principio de Hollywood: "No nos llames, te llamaremos". El flujo del programa está determinado por el framework, no por el programador. El desarrollador registra código (hotspots) y el framework lo invoca cuando corresponde.

Frozen Spot

Frozen Spots

Definen la arquitectura base del sistema: componentes básicos, sus relaciones y el flujo general. Son inalterables — el desarrollador no los modifica ni reimplementa. Ej: el sistema de routing de Flask, el ciclo de vida del request HTTP.

Hot Spot

Hot Spots

Son los puntos de extensión donde el programador "enchufa" su lógica específica. El framework llama a estos puntos en el momento adecuado. Ej: la función decorada con @app.route en Flask.

Extensibilidad

Extensibilidad

Se puede extender un framework por sobreescritura selectiva (override) o añadiendo código para funcionalidades específicas (add-ons). No se puede modificar el código interno del framework.

# Código Flask del Parcial — análisis completo
from flask import Flask, request, jsonify
app = Flask(__name__)

def validar_usuario(datos):            # Lógica de negocio auxiliar (NO es hotspot)
    return "nombre" in datos and "email" in datos

def guardar_en_base(datos):            # Lógica de negocio auxiliar (NO es hotspot)
    print(f"Guardando: {datos}")
    return 42

@app.route("/usuarios", methods=["POST"])   # @app.route registra el hotspot
def crear_usuario():                         # ← HOTSPOT: el framework llama esta función
    datos = request.get_json()              # request: objeto del frozen spot (Flask lo crea)
    if not validar_usuario(datos):
        return jsonify({"error": "datos incompletos"}), 400
    id_nuevo = guardar_en_base(datos)
    return jsonify({"id": id_nuevo}), 201

if __name__ == "__main__":
    app.run(debug=True)                     # Arranca el frozen spot: servidor HTTP

Tema 3

Concurrencia (Sin Actores)

La concurrencia permite que dos o más secuencias de eventos ocurran en períodos superpuestos. No necesariamente en simultáneo (eso sería paralelismo real), sino intercaladas. Los lenguajes deben proveer abstracciones para manejar comunicación, sincronización y atomicidad.

Modelo

Multiprogramación vs. Multiprocesos

Multiprogramación: una sola CPU ejecuta varios programas, las acciones de uno pueden suceder entre dos pasos de otro.
Multiprocesos: múltiples CPUs conectadas, las acciones pueden suceder en simultáneo real.

Problema

Condición de Carrera (Race Condition)

Ocurre cuando el resultado del programa depende del orden de ejecución de los hilos. Ej: dos hilos leen numero=5, ambos lo incrementan a 6, resultado: se pierde un incremento. Solución: exclusión mutua.

Problema

Deadlock (Abrazo Mortal)

Dos o más procesos se bloquean mutuamente, cada uno esperando un recurso que el otro retiene. Ninguno puede continuar. Se evita con políticas de adquisición de locks en orden fijo, o con timeouts.

Solución

Sección Crítica

Fragmento de código donde se accede a recursos compartidos. Se permite un solo proceso a la vez. Las primitivas de exclusión mutua (mutex, semáforos, métodos synchronized) garantizan esto.

Primitivas de Exclusión Mutua

Hardware

Test-and-Set Atómico

Instrucción de hardware que lee y escribe de forma atómica. Se combina con un loop de espera activa (busy-wait) para implementar un mutex básico. Problema: el proceso consume CPU mientras espera.

SO

Semáforo

Evita el busy-wait. Mantiene una cola de procesos en espera; el proceso "duerme" en lugar de girar. El scheduler maneja el semáforo, se inhabilitan interrupciones durante las operaciones.

OOP

Monitor

Concepto de Brinch-Hansen / Hoare. Encapsula datos privados + métodos + política de sincronización. Máximo un proceso ejecuta un método del monitor a la vez. En Java: synchronized es un monitor.

Alto Nivel

Future / Promise

Representa un valor que estará disponible asincrónicamente. No bloquea el hilo principal. Se pueden componer con map, flatMap, sequence. Ideal para tareas I/O-bound.

Código del Parcial: Scala — Threads y Futures

// Código Scala del Parcial — con análisis por fragmento
import scala.concurrent.{Future, Await}
import scala.concurrent.ExecutionContext.Implicits.global
import scala.concurrent.duration._

var contador = 0

// (A) — Manejo explícito de procesos
// Se crea y lanza un Thread nativo del SO
val t = new Thread(() => {
  Thread.sleep(100)
  contador += 1        // ⚠️ Race condition potencial: var mutable compartida
})
t.start()              // Inicia el hilo → ejecución concurrente

// (B) — Ejecución concurrente con Futures
// Se lanzan dos cómputos asincrónicos en paralelo
val f1 = Future { Thread.sleep(300); "resultado 1" }
val f2 = Future { Thread.sleep(200); "resultado 2" }

// (C) — Sincronización + Ejecución concurrente
// Future.sequence combina dos futures en uno solo
// El resultado espera a que AMBOS terminen
val combinado = Future.sequence(List(f1, f2))

// (D) — Sincronización (bloqueo explícito)
// t.join() bloquea el hilo principal hasta que t termine
t.join()

// (E) — Sincronización (bloqueo explícito)
// Await.result bloquea hasta que el Future complete (timeout: 5s)
val resultados = Await.result(combinado, 5.seconds)
println(s"Contador: $contador, Resultados: $resultados")

⚠️

Race Condition en el fragmento A

La variable contador es mutable y compartida. Si el hilo principal accediera a contador antes del t.join(), podría leer un valor inconsistente. Aquí el t.join() en (D) actúa como barrera de sincronización.

Sincronización en Java (contexto teórico)

// Método sincronizado en Java — equivalente al monitor
public synchronized void produce(Object obj) {
    stack.add(obj);
    notify();       // despierta a un consumidor que espera
}

public synchronized Object consume() {
    while (stack.isEmpty()) {
        wait();     // duerme y libera el lock hasta que notify() lo despierte
    }
    return stack.remove(stack.size() - 1);
}

// Estados de un Thread Java:
// New → Ejecutable ↔ Bloqueado → Muerto
// wait() → bloqueado; notify() → ejecutable
📌

¿Por qué el while en el consumidor?

Porque notify() despierta al hilo pero no garantiza que la condición siga siendo verdadera cuando retome (otro hilo pudo consumir el elemento). El bucle while(isEmpty()) { wait(); } rechequea la condición antes de proceder.

Tema 4

Programación Defensiva y Seguridad

"Defend against the impossible, because the impossible will happen." La programación defensiva no solo caza bugs: busca hacer código predecible ante cualquier circunstancia, especialmente las inesperadas, y reducir la superficie de ataque.

Errores

Errores Esperables vs. Prevenibles

Esperables (el entorno los origina): input de usuario inválido, recursos del sistema agotados, fallo de hardware.
Prevenibles (mal funcionamiento del software): argumentos inválidos, valores fuera de rango, valores de retorno no documentados.

Técnica

Diseño por Contrato

Usar precondiciones, postcondiciones e invariantes para asegurar que los datos estén saneados. El código documenta sus supuestos y los verifica. Los assert son una implementación práctica de precondiciones.

Input

Input y Output Seguros

Tres estrategias: Validación (whitelist/blacklist), Escapar datos (sanitizar antes de usar), Canonicalización (normalizar representaciones antes de validar para evitar bypass).

Paradigma

Graceful Degradation vs. Offensive Programming

Defensiva: detecta y tolera errores, el sistema sigue funcionando en modo degradado.
Ofensiva: elimina todas las estrategias de "salvar errores" para que los bugs sean visibles y fáciles de detectar.

Código del Parcial: Asserts y Diseño por Contrato

⚠️

El rol de los Asserts

Los assert son para errores lógicos internos durante el desarrollo — documentan supuestos que nunca deberían violarse si el código es correcto. No son mecanismos de manejo de excepciones para producción, ya que pueden ser desactivados globalmente con -O en Python.

# Código del Parcial — análisis de cada assert
def calcular_promedio(valores):
    assert isinstance(valores, list), "valores debe ser una lista"
    # ↑ Precondición: verifica el tipo antes de ejecutar
    
    assert len(valores) > 0, "la lista no puede estar vacía"
    # ↑ Precondición: evita división por cero
    
    assert all(isinstance(v, (int, float)) for v in valores), \
        "todos los valores deben ser numéricos"
    # ↑ Precondición: valida que todos los elementos sean numéricos
    
    return sum(valores) / len(valores)
    # Si se llega aquí, todas las precondiciones se cumplieron

Canonicalización: Defenderse de Representaciones Inesperadas

🔓

Ejemplo clásico de bypass

Si chequeás que un archivo no sea /etc/passwd, un cracker puede pasar /etc/./passwd — que es el mismo archivo pero con representación diferente. La canonicalización normaliza primero (realpath(), librerías de canonicalización) y luego valida.

# Mal: validar antes de canonicalizar
if path != "/etc/passwd":
    open(path)           # bypass: "/etc/./passwd" pasa la guarda

# Bien: canonicalizar primero, luego validar
import os
canonical = os.path.realpath(path)
if canonical != "/etc/passwd":
    open(canonical)      # seguro

Estrategias: Whitelist vs. Blacklist

Más Segura

Whitelist (Lista Blanca)

Solo se permite lo que está explícitamente autorizado. Todo lo desconocido se rechaza. Más restrictiva pero más segura. Ej: solo aceptar emails con formato válido, rechazar todo lo demás.

Más Permisiva

Blacklist (Lista Negra)

Se bloquea lo conocido como peligroso. Permite todo lo demás. Problema: los atacantes inventan nuevas representaciones que no están en la lista. Menos segura pero más fácil de mantener.

Seguridad en Programación Asistida por IA

🤖

Riesgos del Vibe Coding

Problemas frecuentes al usar IA para programar: alucinaciones (código con errores sutiles que pasan tests normales pero fallan en casos borde), overreliance (no leer ni revisar el código), tipos implícitos ignorados (la IA toma los nombres pero no respeta los tipos), prompt injection y data poisoning. Los errores silenciosos son los más peligrosos.

Tema 5

Lenguajes de Scripting

Los lenguajes de scripting tienen dos grandes antepasados: los intérpretes de comandos (shells) y las herramientas de procesamiento de texto (sed, awk). Evolucionaron hacia Python, Perl, Ruby, JavaScript. Sus características los hacen ideales para automatización y tareas de alto nivel.

Tipos

Tipado Dinámico Flexible

El tipo de una variable solo se comprueba justo antes del uso. En Python, Ruby y PHP, no se declaran tipos. Esto da flexibilidad pero puede causar errores en tiempo de ejecución difíciles de detectar.

Alcance

Reglas de Alcance Simples (o implícitas)

Sin declaraciones explícitas de tipo. En Python: todo es local al bloque donde se hace la asignación. En Perl: por defecto global. En PHP: por defecto local, globals se importan explícitamente.

SO

Fácil Acceso al Sistema Operativo

Los lenguajes de scripting proveen soporte mucho más directo al sistema operativo que lenguajes como Java. Apertura de archivos, pipes, ejecución de comandos del shell, todo con pocas líneas.

Dominio

Construcciones de Alto Nivel

Listas, diccionarios, comprensiones, iteradores — estructuras de datos de alto nivel como parte del lenguaje. Se escribe mucho menos que en Java para lograr lo mismo.

Glosario Bosterizado y Sincronizado

Backtracking
Mecanismo de Prolog: cuando una rama de búsqueda falla, vuelve al último punto de elección y prueba la siguiente alternativa. Es la base del no-determinismo de Prolog.
Unificación
Proceso de igualar dos términos de Prolog encontrando una sustitución de variables que los haga idénticos. Es pattern-matching que determina qué instanciaciones hacer durante la resolución.
Cláusula de Horn
Fórmula lógica con una sola conclusión (cabeza) y un conjunto de premisas: h ← p1, p2, …, pn. Base teórica de Prolog. No todo predicado lógico es expresable como cláusula de Horn.
Cut (!)
Operador de Prolog que fuerza a no revisar alternativas una vez encontrado un resultado. Green cut: no cambia la semántica. Red cut: cambia el comportamiento lógico.
Inversión de Control (IoC)
El principio de Hollywood: "no nos llames, te llamaremos". El framework toma el flujo de la aplicación y ejecuta el código del desarrollador cuando corresponde. Diferencia clave entre framework y librería.
Hotspot
Punto de extensión que expone el framework para que el programador "enchufe" su código personalizado. Ej: @app.route en Flask. El framework invoca estos puntos en el momento adecuado.
Frozen Spot
Código intocable del framework. La arquitectura interna base: routing HTTP, ciclo de vida de requests, manejo de respuestas. El desarrollador no lo modifica ni reimplementa.
Exclusión Mutua (Mutex)
Mecanismo que asegura que solo un hilo a la vez pueda acceder a una sección crítica o recurso compartido. Evita condiciones de carrera. Implementado con semáforos, monitores o synchronized.
Sincronización (Sync)
Coordinación de la ejecución de múltiples hilos para que operen en el orden correcto. Ej: join() espera que un hilo termine; Await.result() espera que un Future complete.
Condición de Carrera (Race Condition)
Fallo donde el resultado del programa concurrente depende del orden de ejecución de los hilos. Ocurre cuando múltiples hilos leen/escriben datos compartidos sin sincronización.
Deadlock (Abrazo Mortal)
Situación donde dos o más procesos se bloquean eternamente, cada uno esperando que el otro libere un recurso. Ninguno puede avanzar. Se previene con orden fijo de adquisición de locks.
Monitor / Semáforo
Monitor: encapsula datos privados + métodos + política de sincronización (Brinch-Hansen). Semáforo: cola de procesos durmientes, evita busy-wait. Ambos implementan exclusión mutua.
Future / Promise
Representa un valor que estará disponible asincrónicamente. No bloquea el hilo principal. Se componen con map, flatMap, sequence. Ideal para tareas I/O concurrentes.
Diseño por Contrato
Técnica defensiva que usa precondiciones (antes de ejecutar), postcondiciones (después) e invariantes (siempre) para documentar y verificar supuestos. Los assert son una implementación práctica.
Programación Ofensiva
Elimina todas las estrategias de "salvar errores" para que los bugs sean visibles inmediatamente. Opuesto a graceful degradation. El objetivo es descubrir malfuncionamientos, no tolerarlos.
Canonicalización
Normalizar representaciones de datos antes de validarlos. Evita bypass de seguridad (ej: /etc/./passwd es el mismo archivo que /etc/passwd). Siempre canonicalizar antes de aplicar whitelist/blacklist.
Tipado Dinámico
El tipo de una variable se comprueba justo antes del uso, no en la declaración. Propio de lenguajes de scripting (Python, Ruby, PHP). Flexible pero puede producir errores difíciles de detectar.
Completitud de Prolog
Prolog es incompleto: devuelve solo cosas verdaderas, pero no puede probar todo lo que es verdadero. Además, el orden de cláusulas puede causar bucles infinitos (recursión a izquierda).
