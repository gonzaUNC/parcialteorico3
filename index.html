02 · Fundamentos

Sintaxis & Semántica

Sintaxis

La forma del lenguaje

Describe qué se puede escribir legalmente en el lenguaje. Se formaliza con gramáticas independientes de contexto (EBNF).

Semántica

El significado del lenguaje

Describe qué efectos tiene lo escrito en la máquina. Tipos: operacional, denotacional, lambda cálculo.

EBNF

Gramáticas formales

Estándar de facto para gramáticas libres de contexto. No son Turing-completas, por eso necesitan análisis semántico adicional en el compilador.

Gramática: Ejemplo clásico de expresiones

Gramática EBNF — Expresiones aritméticas<expresion> → numero
<expresion> → ( <expresion> )
<expresion> → <expresion> + <expresion>
<expresion> → <expresion> - <expresion>
<expresion> → <expresion> * <expresion>
<expresion> → <expresion> / <expresion>
⚠️

Limitación de gramáticas libres de contexto

No pueden expresar fácilmente: que una variable sea declarada antes de usarse, ni asignación múltiple como (foo, bar) = 1, 2 en Python. El compilador resuelve esto en el análisis semántico.

Tipos de semántica

TipoDescripciónReferencia
Operacional Describe cómo cada paso del cálculo se lleva a cabo. Usa un modelo simplificado de la máquina. La más usada en este curso. Mitchell 4.1
Lambda Cálculo Formalización de Church. Primera semántica de LISP (McCarthy). Base del paradigma funcional. Mitchell 4.2
Denotacional De Strachey & Scott. Mapea programas a funciones matemáticas. Muy formal y rigurosa. Mitchell 4.3
📐

Tesis de Church-Turing

Las funciones computables son exactamente las funciones que se pueden calcular usando un dispositivo mecánico dada tiempo y espacio ilimitados. Funciones no computables famosas: Halting Problem, Complejidad de Kolmogorov.

03 · Implementación

Compiladores & Variables

Estructura de un compilador

Paso 1

Análisis Léxico

Tokeniza el código fuente. Convierte el stream de caracteres en tokens (palabras clave, identificadores, literales).

Paso 2

Análisis Sintáctico (Parser)

Usa la gramática para construir el árbol de derivación (AST). Verifica que la forma del programa sea correcta.

Paso 3

Análisis Semántico

Resuelve tipos, alcance, declaraciones previas. Lo que la gramática libre de contexto no puede capturar.

Paso 4

Generación de Código

Traduce el AST a código máquina. Reemplaza los nombres simbólicos de variables con ubicaciones de memoria reales.

Variables: L-valor y R-valor

Una variable tiene dos aspectos diferenciados que son clave para entender el pasaje de parámetros y el aliasing:

ConceptoDefiniciónEjemplo
L-valor La ubicación en memoria de la variable (dirección). El "dónde está". En x := y+3, usamos el L-valor de x (su dirección).
R-valor El valor almacenado en la ubicación (contenido). El "qué contiene". En x := y+3, usamos el R-valor de y (su contenido).
Aliasing Cuando dos identificadores distintos referencian la misma ubicación en memoria. Ocurre con pasaje por referencia de la misma variable, o con punteros en C.

Celdas de referencia en ML

ML — Operaciones con celdas de referenciaref v       (* Crea una celda que contiene v *)
!r          (* Lee el valor de la celda r *)
r := v      (* Escribe v en la celda r *)

(* Ejemplo completo *)
val x = ref 42    (* x : int ref *)
val y = !x          (* y = 42 *)
x := 100            (* x ahora apunta a 100 *)
04 · Tipos

Tipos de Datos

Estático

Tipado estático

Los tipos se verifican en tiempo de compilación. Errores detectados antes de ejecutar. Ej: Java, C++, Haskell, ML.

Dinámico

Tipado dinámico

Los tipos se verifican en tiempo de ejecución. Más flexible pero puede fallar en runtime. Ej: Python, Ruby, JavaScript.

Fuerte

Tipado fuerte

No se permiten conversiones implícitas inseguras. Los tipos no se mezclan sin operación explícita. Ej: Python, Haskell.

Débil

Tipado débil

Permite conversiones implícitas (casteos automáticos). Puede generar comportamiento inesperado. Ej: Perl, C, JavaScript.

Inferencia de tipos en ML

ML puede deducir el tipo de una expresión sin que el programador lo declare explícitamente. El compilador resuelve un sistema de ecuaciones de tipos.

ML — Inferencia de tipos(* ML infiere: aplicar : ('a -> 'b) -> 'a -> 'b *)
fun aplicar(f, x) = f(x)

(* ML infiere: a : (int * int) -> bool *)
fun a(x, y) = (x > 2) orelse (y < 10)

(* Duck typing en Crystal: compila si el tipo tiene el método *)
(* Si Duck quacks y Goose quacks: tipo inferido = Duck | Goose *)
(* Error en compile-time, NO en runtime (a diferencia de Python) *)
🦆

Duck Typing

"Si camina como un pato y grazna como un pato, entonces es un pato." El tipo se determina por los métodos disponibles, no por la declaración explícita. Lenguajes dinámicos como Python lo usan extensivamente.

05 · Estructura

Bloques & Activation Records

🍝

Código Spaghetti

Código con saltos GOTO arbitrarios que lo hacen incomprensible. Ejemplo: Fortran histórico. La estructura de bloques fue creada para resolverlo.

Propiedades de los lenguajes estructurados por bloques

Propiedad 1

Declaración local

Variables se declaran dentro de bloques. Solo son visibles en ese bloque y los anidados (variables locales).

Propiedad 2

Anidamiento no solapado

Los bloques pueden anidarse pero no solaparse parcialmente. Si comparten código, uno debe estar enteramente dentro del otro.

Propiedad 3

Memoria dinámica

Al entrar al bloque se asigna memoria; al salir, se libera. Esto permite recursión y variables independientes por llamada.

Activation Record (Stack Frame)

Estructura de datos apilada en el stack de ejecución cuando el programa entra a un bloque. Contiene todo lo necesario para esa llamada.

Contenido típico de un Activation Record (x86)┌─────────────────────────────────┐
│  Argumentos recibidos            │  ← escritos por quien llama
│  Variables locales               │  ← declaradas en este bloque
│  Resultados intermedios          │  ← cálculos temporales
│  Control Link  ──────────────►   │  ← apunta al AR anterior (quien llamó)
│  Access Link   ──────────────►   │  ← apunta al AR del bloque contenedor
│  Dirección de retorno            │  ← adónde volver al desapilar
└─────────────────────────────────┘
¿Por qué no hay recursión en Fortran histórico? ▼
En Fortran antiguo, todas las variables (incluyendo parámetros de subrutinas) tenían una ubicación fija en memoria. Si P llama a Q, Q llama a R, y R intenta llamar de nuevo a P, la segunda llamada pisaría los valores de la primera llamada. La estructura de bloques y los activation records en el stack resolvieron esto completamente.
Stack Overflow y Tail Call Optimization (TCO) ▼
Muchas llamadas recursivas apilan muchos activation records. Si la recursión es muy profunda, se agota el stack (stack overflow). La optimización de recursión a la cola reutiliza el mismo activation record para llamadas tail-recursive, manteniendo el stack en tamaño constante. Haskell, Scala, y Scheme lo hacen automáticamente.

Ejemplo de ejecución de bloques

Traza de la pila — bloques anidados// Código:
{ int x = 0;
  int y = x + 1;
  { int z = (x + y) * (x - y); }
}

// Estado de la pila:
// Entrada bloque exterior:  [AR: x=0, y=1]
// Entrada bloque interior:  [AR: x=0, y=1] ← [AR: z=?]
// Salida bloque interior:   [AR: x=0, y=1]
// Salida bloque exterior:   []
06 · Control

Pasaje de Parámetros

Los parámetros formales son los de la declaración de la función. Los parámetros reales son los valores efectivos en una llamada concreta.

MecanismoSe pasaEfectoLenguajes
Por valor R-valor (copia del contenido) Asignaciones dentro no afectan al bloque que llama C, Java, ML
Por referencia L-valor (dirección de memoria) Asignaciones dentro sí afectan al bloque que llama. Aliasing posible. Perl (default), C++ (&)
Por valor-resultado Copia; al terminar, copia de vuelta Como referencia pero sin aliasing durante ejecución. Resultado depende del compilador. Poco usado hoy
Por nombre Expresión sin evaluar (lazy) Se reevalúa cada vez que se usa. Útil si el argumento no siempre se usa. ALGOL 60, Scala (=>)
Por necesidad Expresión lazy + memoización Como por nombre, pero se evalúa una sola vez y se memoriza el resultado. Haskell, R
Call-by-sharing Referencia a objeto (valor) Mutaciones del objeto son visibles; reasignación de la variable no. Java, Python, Ruby, JS

Por valor vs. por referencia — Código comparado

ML — Diferencia semántica(* PASAJE POR VALOR — x es una copia local *)
fun f(z : int) = let x = ref z in x := !x + 1; !x end
(* Si y=0, f(!y) = 1, pero y sigue siendo 0 *)

(* PASAJE POR REFERENCIA — x es un alias de la variable real *)
fun f(x : int ref) = (x := !x + 1; !x)
(* Si y=0, f(y) = 1, y ahora y también es 1 *)
🧠

Truco para el parcial

Para determinar el tipo de pasaje: si dos llamadas f(x, x) con los mismos argumentos dan resultados diferentes, hay aliasing → es por referencia. Si no hay diferencia, es por valor.

07 · Control

Alcance Estático vs. Dinámico

Léxico / Estático

Alcance Estático

Una variable global se refiere al identificador en el bloque contenedor más cercano en el texto del programa. Determinado en tiempo de compilación. La mayoría de lenguajes modernos.

Dinámico

Alcance Dinámico

Una variable global se refiere al identificador del activation record más reciente en el stack en tiempo de ejecución. LISP antiguo, Bash, TeX/LaTeX, macros.

Ejemplo clásico: diferencia visible

Pseudo-código — Alcance estático vs. dinámicoint x = 1;

function g(z) = x + z;      // x libre en g

function f(y) = {
  int x = y + 1;             // x local en f, x=4 si y=3
  return g(y * x);           // llama g(12)
}

f(3);

// En g, x+z con z=12:
// Alcance ESTÁTICO → x = 1 (bloque exterior del texto)  → resultado: 13
// Alcance DINÁMICO → x = 4 (activation record más reciente)  → resultado: 16

Access Links — Implementación del alcance estático

Para implementar alcance estático, cada activation record tiene un access link (puntero al AR del bloque contenedor en el texto). Es distinto al control link (que apunta al AR de quien llamó).

LinkApunta a...Propósito
Control Link AR del bloque que realizó la llamada Saber adónde volver al desapilar (flujo de control)
Access Link AR del bloque contenedor en el texto Encontrar variables libres con semántica léxica
08 · Control

Alto Orden & Clausuras

⭐

Funciones de primera clase

Un lenguaje tiene funciones de primera clase si pueden: (1) declararse en cualquier alcance, (2) pasarse como argumentos, (3) devolverse como resultado de funciones. ML, Haskell, Python, JavaScript las tienen; C no.

Clausura (Closure)

En un lenguaje con alcance estático y funciones de primera clase, un valor de función se representa como una clausura: par formado por un puntero al código de la función + puntero al activation record de su entorno léxico.

ML — Función map: primer orden superior clásicofun map(f, nil)   = nil
  | map(f, x::xs) = f(x) :: map(f, xs)

(* f es una función de primera clase *)
(* Ejemplo: incrementar todos los elementos *)
map(fn x => x + 1, [1, 2, 3])  (* = [2, 3, 4] *)
Problema "Downward Funarg" — ¿Por qué C no tiene clausuras? ▼
Cuando f se pasa como argumento a g, y dentro de g se llama a f, necesitamos el access link del entorno donde f fue declarada. C prohíbe funciones anidadas para evitar este overhead. C++ moderno usa lambdas que capturan el entorno explícitamente.
Recursión a la cola (Tail Call Optimization) ▼
Una función es recursiva de cola si todas las llamadas recursivas son la última operación (no hay cálculo posterior al retorno). El compilador puede reutilizar el mismo activation record, convirtiendo recursión en iteración eficiente en memoria.
fun factcola(n, a) = if n <= 1 then a else factcola(n-1, n*a)
Esta es recursiva de cola. fact(n) = n * fact(n-1) NO lo es (hay multiplicación después del retorno).

10 · Paradigma

Paradigma Funcional

Imperativo Funcional
Expresa cómo calcular (secuencias de pasos) Expresa qué calcular (funciones matemáticas)
Variables mutables y estado compartido Inmutabilidad: los datos no cambian
Loops (for, while) Recursión y funciones de orden superior
Efectos secundarios explícitos Funciones puras: misma entrada → misma salida

Propiedades valiosas del paradigma funcional

Composicionalidad

Funciones como bloques

Las funciones se componen como piezas de Lego. El output de una es el input de la siguiente.

Transparencia referencial

Sin efectos secundarios

Una expresión siempre evalúa al mismo valor. Facilita razonamiento formal, testing y optimización.

Concurrencia declarativa

Paralelismo natural

Sin estado mutable compartido → sin condiciones de carrera. Computación paralela más sencilla y segura.

ML — Expresión funcional vs. imperativa(* Imperativo: expresa CÓMO *)
let suma_imperativa = ref 0
for i = 1 to 10 do suma_imperativa := !suma_imperativa + i done

(* Funcional: expresa QUÉ *)
let suma_funcional = List.fold_left (+) 0 [1..10]
⚠️

Problemas naturalmente no declarativos

Algunas tareas son inherentemente imperativas: operaciones I/O, interacción con hardware, generadores de números pseudo-aleatorios. Los lenguajes funcionales usan mónadas (Haskell) u otras abstracciones para manejar estos efectos controladamente.

11 · Paradigma

Orientación a Objetos

Pilar 1

Encapsulación

Los objetos ocultan su implementación interna. Solo se expone una interfaz pública. Protege el estado y reduce el acoplamiento.

Pilar 2

Herencia

Una clase puede heredar atributos y métodos de una clase padre. Permite reutilización de código y jerarquías de tipos.

Pilar 3

Polimorfismo

Una misma interfaz funciona para diferentes tipos. Despacho dinámico (late binding) en métodos virtuales.

Extra

Abstracción

Las clases abstractas/interfaces definen contratos sin implementación. Permiten separar interfaz de implementación.

C++ vs Java — Diferencias clave

CaracterísticaC++Java
Herencia múltiple Sí (con sus problemas) Solo de interfaces
Gestión de memoria Manual (new/delete) Automática (Garbage Collector)
Métodos virtuales Explícitos con virtual Todo método es virtual por default
Pasaje de parámetros Por valor o referencia (&) Call-by-sharing (referencias a objetos)
🔗

Relación con clausuras

Una clausura y un objeto son conceptualmente similares: ambos combinan datos y código. Un objeto es como una clausura con múltiples métodos; una clausura es como un objeto con un solo método.

12 · Paradigma

Programación Lógica — Prolog

Primitiva básica

Relaciones (predicados)

En lugar de F(x)=y (función), se declaran relaciones: R(x,y) — "se da la relación R entre x e y". El motor de Prolog busca si esa relación se puede satisfacer.

Origen

Alain Colmerauer, 1972

Creado para procesamiento del lenguaje natural. "Programmation en logique". Aplicado en IA, bases de datos, sistemas expertos.

Ventaja clave

Más poderoso que SQL

Puede usar recursión para consultas de grafo. Ej: encontrar vuelos con escala, no solo directos. SQL no puede hacer consultas recursivas de manera natural.

Bloques de construcción de Prolog

BloqueSintaxisSignificado
Hecho (Fact) padre(tom, bob). Una relación que se declara verdadera sin condiciones.
Regla (Rule) abuelo(X,Z) :- padre(X,Y), padre(Y,Z). X es abuelo de Z si X es padre de Y y Y es padre de Z.
Consulta (Query) ?- abuelo(tom, ann). ¿Es tom abuelo de ann? Prolog responde true/false o unifica variables.
Variable X, Y, Who (mayúscula) Se unifica con valores durante la búsqueda. Una variable libre puede tomar cualquier valor.
Átomo tom, ana, okc (minúscula) Constante simbólica. No puede cambiar de valor.

Unificación

El mecanismo central de Prolog. Dos términos unifican si existe una sustitución de variables que los hace idénticos. Es la forma en que Prolog "llena" las variables.

Prolog — Ejemplos de unificación% Unificación exitosa:
?- X = 3.          % X = 3  (X se liga a 3)
?- f(X, b) = f(a, b).   % X = a  (X se liga a a)
?- [H|T] = [1,2,3].     % H = 1, T = [2,3]

% Unificación fallida:
?- f(a) = f(b).    % false  (a ≠ b)
?- X = 1, X = 2.   % false  (X no puede ser 1 y 2 a la vez)

Backtracking

Cuando un subobjetivo falla, Prolog retrocede al punto de elección anterior y prueba la siguiente alternativa. Es el motor de búsqueda automático de Prolog.

Prolog — Cómo funciona el backtrackingcolor(rojo). color(verde). color(azul).

% ?- color(X), X = verde.
% 1. Prueba color(X) con X = rojo → ok, luego rojo = verde → FALLA
% 2. Backtrack → prueba color(X) con X = verde → ok, verde = verde → ok
% Resultado: X = verde

Negación: not/1

⚠️

not(X) en Prolog = "No se puede probar X"

not(Goal) es verdadero si Prolog no puede demostrar Goal. No es negación lógica clásica — es Negation as Failure. Si Goal contiene variables sin ligar, el resultado puede ser inesperado.

Prolog — Base de datos con reglas encadenadas% Hechos
especialista(ana, traumatologia).
especialista(elena, traumatologia).
especialista(clara, otorrinolaringologia).
deriva(clara, ana).
deriva(esteban, ana).

% Reglas
departamento(X, Y) :- especialista(X, A), especialista(Y, A).
jefe(X, Y)         :- deriva(X, Y), departamento(X, Y).
equipo(X, Y)       :- departamento(X, Y), not(jefe(X, Y)).

% Consultas:
?- departamento(ana, elena).   % true (ambas en traumatologia)
?- jefe(clara, ana).           % false (clara es de otorrino, no mismo depto)
?- equipo(elena, ana).         % ← ver ejercicio resuelto →
🔍

Aplicaciones típicas de Prolog

Procesamiento de lenguaje natural · Razonamiento automático · Probadores de teoremas · Búsqueda en bases de datos relacionales · Sistemas expertos · Problemas de búsqueda (N-Reinas, Sudoku).

12b · Ejercicio Resuelto

Prolog: ¿Cómo resolver una consulta paso a paso?

📋

Base de conocimiento del ejercicio (del parcial)

Se trabaja con médicos, especialidades, derivaciones y relaciones de equipo/jefatura. La pregunta es: ¿Es equipo(elena, ana) verdadero?

Base de conocimiento completa% Hechos
especialista(ana, traumatologia).
especialista(elena, traumatologia).
especialista(clara, otorrinolaringologia).
especialista(esteban, traumatologia).
deriva(clara, ana).
deriva(clara, elena).
deriva(esteban, ana).

% Reglas
departamento(X, Y) :- especialista(X, A), especialista(Y, A).
jefe(X, Y)         :- deriva(X, Y), departamento(X, Y).
equipo(X, Y)       :- departamento(X, Y), not(jefe(X, Y)).

Paso a paso: ?- equipo(elena, ana).

1
Buscar definición de equipo/2

La única cláusula es: equipo(X,Y) :- departamento(X,Y), not(jefe(X,Y)).
Prolog unifica X=elena, Y=ana y genera dos subobjetivos.

2
Resolver departamento(elena, ana)

Regla: departamento(X,Y) :- especialista(X,A), especialista(Y,A).
Unifica X=elena, Y=ana. Busca especialista(elena, A).
Encuentra: especialista(elena, traumatologia) → A = traumatologia.
Luego busca especialista(ana, traumatologia) → ✓ TRUE.

3
Resolver not(jefe(elena, ana))

Prolog intenta probar jefe(elena, ana) para luego negar el resultado.
Regla: jefe(X,Y) :- deriva(X,Y), departamento(X,Y).
Necesita deriva(elena, ana). Busca en los hechos...

4
Verificar deriva(elena, ana)

Los hechos de deriva son: deriva(clara, ana), deriva(clara, elena), deriva(esteban, ana).
Ninguno tiene elena como primer argumento.
→ deriva(elena, ana) es FALSE.
→ jefe(elena, ana) es FALSE.

5
Resultado final

not(jefe(elena, ana)) = not(FALSE) = TRUE
Ambos subobjetivos son true → equipo(elena, ana) = TRUE ✓

Tabla resumen de subobjetivos (como en el parcial)

SubobjetivoResultado¿Por qué?
especialista(elena, traumatologia) TRUE — necesario Es un hecho en la base. Necesario para demostrar departamento(elena,ana).
departamento(elena, ana) TRUE — necesario Ambas son especialistas en traumatologia. Es el primer subobjetivo de equipo/2.
departamento(ana, elena) No se comprueba directamente equipo(elena,ana) comprueba departamento(elena,ana), no al revés. Aunque sería true, no es lo que se evalúa.
deriva(elena, ana) FALSE — no existe Se intenta para probar jefe(elena,ana). No existe en la base → jefe falla → not(jefe) es true.
🧠

Estrategia para resolver ejercicios de Prolog en el parcial

1. Escribí la consulta y buscá su definición en las reglas.
2. Sustituí las variables con los valores concretos de la consulta.
3. Resolvé cada subobjetivo de izquierda a derecha.
4. Para not(X): intentá probar X. Si X falla → not(X) es true.
5. Si un subobjetivo falla, hacé backtracking al punto de elección anterior.

13 · Paradigma

Programación Concurrente

Dos o más secuencias de eventos que ocurren en paralelo. Hay dos modelos principales:

Modelo 1

Multiprogramación

Una computadora ejecuta varios programas. Cada uno funciona secuencialmente. Las acciones de un programa pueden suceder entre dos pasos de otro (interleaving).

Modelo 2

Multiproceso

Dos o más procesadores conectados. Las acciones pueden suceder simultáneamente. Comunicación entre procesadores necesaria.

La promesa (y los retos) de la concurrencia

PromesaReto
Velocidad: n procesadores → t/n tiempo? Comunicación: ¿cómo recibir/enviar info?
Disponibilidad: si un proceso está ocupado, otro ayuda Sincronización: ¿cómo esperar a otro proceso?
Distribución: procesadores en distintos lugares colaboran Atomicidad: garantizar que una operación no quede a mitad

Construcciones que esperamos de un lenguaje concurrente

Hilos

Threads como valores

Los hilos son ciudadanos de primera clase. Se pueden pasar a funciones, crear como resultado de una llamada.

Comunicación

Canales y mensajes

Comunicación síncrona o canales con buffer asíncrono que preservan el orden de mensajes.

Control

Exclusión mutua

Cerrojos (locks), semáforos, monitores. La atomicidad verdadera es menos común en los lenguajes.

✨

Concurrencia declarativa

En lenguajes funcionales puros (sin estado mutable compartido), la concurrencia es mucho más segura. No hay condiciones de carrera sobre datos inmutables. Scala, Erlang y Haskell aprovechan esto.

15 · Herramientas

Frameworks de Programación

🏗️

¿Qué es un Framework?

Una abstracción de software que provee funcionalidades genéricas que se pueden instanciar para aplicaciones específicas. Es la evolución del boilerplate. Método estándar para desplegar aplicaciones.

Concepto clave

Inversión de Control (IoC)

"No nos llame, nosotros lo llamaremos" — Principio de Hollywood. El flujo del programa está determinado por el framework, no por el programador.

Concepto clave

Frozen Spots vs Hot Spots

Frozen: arquitectura fija, inalterables. Hot: partes donde el programador añade código específico del proyecto.

vs. Librerías

Diferencia esencial

Con una librería, tu código la llama. Con un framework, el framework llama a tu código. La diferencia es quién tiene el control del flujo.

Arquitectura MVC

MVC — Model · View · ControllerModel       → Datos y lógica de negocio
View        → Presentación e interfaz de usuario
Controller  → Intermediario; recibe input y actualiza Model/View

Desventajas de los Frameworks

ProblemaDescripción
Code bloat Código muy voluminoso para aplicaciones pequeñas. Se mantienen funcionalidades deprecadas.
Curva de aprendizaje Tiempo de aprendizaje largo. Solo rinde si se usa para más de un proyecto.
No modificable No se puede cambiar el código del framework. Solo se extiende mediante Override o add-ons.
16 · Calidad

Programación Defensiva

🛡️

Principio fundamental

"Defend against the impossible, because the impossible will happen." — Principalmente causado por input de usuario, cambios en el entorno, y ataques de inyección de código.

Errores esperables

Del entorno

Input de usuario inválido · Recursos del sistema agotados · Fallo de hardware. Son previsibles; deben manejarse.

Errores prevenibles

Del software

Argumentos de función inválidos · Valor fuera de rango · Valor de retorno no documentado. Son bugs; deben eliminarse.

Técnicas

Herramientas

Reuso de software inteligente · Canonicalización de inputs · Guardas (if (inesperado) throw) · Validación con whitelist/blacklist.

Software Legacy

⚰️

Problemas del software heredado

Diseñado para ASCII pero ahora recibe UTF-8 · Compilado para 32-bit pero ejecutado en 64-bit · Nunca tuvo conectividad de red, ahora sí · Vulnerable a code injection porque no era un problema conocido entonces.

Referencia

Glosario

Activation Record
Estructura de datos en el stack de ejecución creada al entrar a un bloque. Contiene variables locales, control link, access link y dirección de retorno. También llamado stack frame.
Aliasing
Cuando dos identificadores distintos referencian la misma ubicación en memoria. Ocurre con pasaje por referencia de la misma variable o con punteros. Puede generar efectos secundarios inesperados.
Alcance Estático (Léxico)
Una variable libre se refiere al identificador del bloque contenedor más cercano en el texto del programa. Determinado en compilación. La mayoría de lenguajes modernos.
Alcance Dinámico
Una variable libre se refiere al identificador del activation record más reciente en el stack en tiempo de ejecución. Lisp antiguo, Bash, TeX.
Access Link
Puntero dentro de un activation record que apunta al activation record del bloque contenedor en el texto del programa. Implementa el alcance estático. Distinto al control link.
Clausura (Closure)
Valor de función en lenguajes con alcance estático y funciones de primera clase. Par de: puntero al código de la función + puntero al activation record de su entorno léxico.
Control Link
Puntero dentro de un activation record que apunta al activation record del bloque que realizó la llamada. Permite volver al código anterior al desapilar.
Call-by-Value
Se pasa el R-valor (contenido) del parámetro. Se crea una copia local en el AR de la función llamada. Asignaciones internas no afectan al bloque que llama. El más común.
Call-by-Reference
Se pasa el L-valor (dirección) del parámetro. La función puede modificar directamente la variable del bloque que llama. Genera aliasing. Perl por defecto.
Call-by-Name
El argumento no se evalúa antes de la llamada. Se sustituye directamente en el cuerpo. Se re-evalúa cada vez que aparece. ALGOL 60, Scala.
Call-by-Need (Lazy)
Versión memoizada de call-by-name. Se evalúa la primera vez que se usa y se almacena. Haskell. Evita recomputación y permite estructuras de datos infinitas.
Call-by-Sharing
Se pasa por valor una referencia a un objeto. Mutaciones del objeto son visibles en el llamador; reasignación de la variable no. Java, Python, Ruby, JavaScript.
Gramática Libre de Contexto (EBNF)
Herramienta formal para describir la sintaxis de lenguajes. Estándar EBNF. No puede expresar restricciones como "variable declarada antes de usarse". El análisis semántico complementa.
Inversión de Control (IoC)
Principio de Hollywood: "No nos llame, nosotros lo llamaremos." El framework controla el flujo, no el programador. Diferencia clave entre framework y librería.
L-valor / R-valor
L-valor: ubicación en memoria (dirección) de la variable. R-valor: valor almacenado en esa ubicación. En ML, tienen tipos distintos. En C, el L-valor puede perderse sin advertencia.
MVC
Model-View-Controller. Patrón arquitectónico para separar datos (Model), presentación (View) y lógica de interacción (Controller). Muy usado en frameworks web.
Polimorfismo
Una misma interfaz funciona para diferentes tipos. En OOP: despacho dinámico (late binding). El método llamado se determina en runtime según el tipo real del objeto.
Recursión de Cola (TCO)
Una función es recursiva de cola si la llamada recursiva es la última operación. Los compiladores pueden reutilizar el AR, evitando stack overflow. Haskell, Scala, Scheme.
Semántica Operacional
Describe cómo cada paso de un cálculo se ejecuta en un modelo simplificado de la máquina. Define el significado de un programa como secuencias de pasos computacionales. La más usada en el curso.
Sintaxis / Semántica
Sintaxis: la forma del lenguaje (qué se puede escribir legalmente). Semántica: el significado (qué efectos tiene lo escrito). Todo lenguaje tiene ambas dimensiones.
Stack Overflow
Error cuando el stack de ejecución se llena por demasiados activation records (recursión muy profunda sin TCO). Solucionado por compiladores que implementan Tail Call Optimization.
Tesis de Church-Turing
Las funciones computables son exactamente las calculables por un dispositivo mecánico con tiempo y espacio ilimitados. Cualquier función con algoritmo es computable. Formaliza la noción intuitiva de algoritmo.
Tipado (clasificación)
Estático: tipos en compilación. Dinámico: tipos en runtime. Fuerte: sin conversiones implícitas inseguras. Débil: casteos automáticos. Duck typing: "si tiene el método, sirve".
Prolog / Programación Lógica
La primitiva básica es la relación/predicado R(x,y), no la función F(x)=y. El motor de Prolog hace backtracking automático para encontrar todas las soluciones que satisfacen las reglas.
